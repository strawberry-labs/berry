## Product decision

Berry Apps should be built **on top of Agent Skills**, not as a competing
execution system.

- An Agent Skill remains the portable instruction and resource package.
- A Berry App adds the product contract that a skill does not provide:
  typed inputs, declared outputs, a catalog, versioning, access control,
  sandbox policy, run history, and a purpose-built form/result UI.
- A Berry App package is an Agent Skills-compatible directory containing
  `SKILL.md`, optional `scripts/`, `references/`, and `assets/`, plus a
  Berry-specific `berry.app.yaml` manifest.
- App runs reuse Berry's existing task/session, model-governance, budget,
  approval, sandbox, file, artifact, audit, and usage systems.

This split preserves portability while giving Berry a differentiated,
enterprise-ready application surface.

## Formal feature definition

### Feature name

**Berry Apps**

### User promise

An administrator can package and publish a repeatable AI workflow as an app.
A member opens that app, completes a small form or uploads files, runs it, and
receives a message, downloadable files, or both without writing a prompt.

### Personas

- **App administrator**: imports, reviews, tests, versions, publishes, and
  archives apps.
- **App user**: discovers authorized published apps, runs them, handles any
  approval requests, and revisits prior results.
- **Platform operator**: configures the model, storage, budgets, network, and
  sandbox policies that bound every app run.

### Core lifecycle

1. An administrator imports a `.skill` ZIP containing `SKILL.md`,
   `berry.app.yaml`, and optional resources.
2. Berry validates paths, package limits, the Agent Skill frontmatter, the app
   manifest, requested tools, and output contract.
3. Berry stores a mutable draft version. The administrator can run that exact
   draft in test mode.
4. Publishing makes the version immutable and sets it as the app's current
   version.
5. A member sees only published apps allowed by tenant permissions and
   resource ACLs.
6. The member supplies manifest-defined inputs. Berry validates the values and
   ownership of uploaded file IDs.
7. Berry creates a normal task and session, forces explicit activation of the
   versioned skill, applies the app's tool/network/sandbox limits, and uses the
   existing governed agent-turn pipeline.
8. The final assistant message and persisted output files appear in the run
   result, the underlying task, run history, and the existing file library.
9. Historical runs remain tied to their immutable version even after a newer
   version is published.

### MVP input and output contract

Input controls:

- `text`
- `textarea`
- `file` (one file per field in MVP)
- `select`
- `boolean`

Output modes:

- `message`
- `files`
- `message-and-files`

The MVP intentionally does not implement a general JSON Schema renderer,
arbitrary nested objects, repeaters, scheduled runs, marketplace distribution,
or a visual workflow builder.

### Success measures

- A user can start a published app run in under three interactions after
  opening the app.
- Every run is attributable to a tenant, user, app, immutable version, task,
  and session.
- A package cannot obtain a tool, network capability, file, or tenant resource
  beyond the intersection of its manifest and organization policy.
- Failed runs expose a safe actionable error and retain enough metadata for an
  administrator to diagnose them without logging user file contents.

## Findings considered and rejected

- **Replace Agent Skills with Apps**: rejected. It would duplicate Berry's
  existing skill activation, progressive disclosure, resources, and portable
  package format.
- **Expose raw skills directly as the Apps catalog**: rejected. Skills do not
  define typed forms, output contracts, version immutability, run history, or
  app-specific permissions.
- **Store apps as organization capabilities**: rejected. The current
  organization-skill path stores `SKILL.md` content but materializes
  `resources: []`; it also has no version or run model.
- **Execute app inference in the BullMQ worker for the MVP**: rejected. The
  current governed runtime, session host, file registration, and approval
  stream are API-local. Moving them would duplicate security-sensitive
  plumbing. Durable cross-process execution is a later architecture project.
- **Trust `allowed-tools` from Agent Skills frontmatter as enforcement**:
  rejected. Berry parses the field but does not currently enforce it. The
  Berry manifest's allowlist must be applied to the actual runtime tool set.
