# Berry Desktop Release

Berry Desktop packages through Tauri from `apps/desktop`.

Required release secrets:

- `APPLE_CERTIFICATE`
- `APPLE_CERTIFICATE_PASSWORD`
- `APPLE_SIGNING_IDENTITY`
- `APPLE_ID`
- `APPLE_PASSWORD`
- `APPLE_TEAM_ID`
- `WINDOWS_CERTIFICATE`
- `WINDOWS_CERTIFICATE_PASSWORD`
- `TAURI_SIGNING_PRIVATE_KEY`
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`
- `TAURI_UPDATER_PUBLIC_KEY`
- `TAURI_UPDATER_ENDPOINT`

`TAURI_SIGNING_PRIVATE_KEY` is the private key generated by the Tauri signer
and must stay in the release secret manager. `TAURI_UPDATER_PUBLIC_KEY` is the
matching public key and may live in repository or organization variables.
`TAURI_UPDATER_ENDPOINT` is the signed updater feed URL, normally:

```text
https://github.com/<owner>/<repo>/releases/latest/download/latest.json
```

Local verification before a release:

```bash
pnpm install --frozen-lockfile
pnpm check:desktop:full
```

Release jobs also run:

```bash
pnpm check:desktop:full
pnpm build
node scripts/verify-release-prereqs.mjs --platform=darwin
node scripts/verify-release-prereqs.mjs --platform=win32
node scripts/verify-release-prereqs.mjs --platform=linux
```

Tagged releases use `desktop-v*` tags and attach the generated Tauri bundle artifacts to GitHub Releases.
The release job also renders `latest.json` from the signed updater artifacts
and publishes it to the same GitHub Release. The packaged app bakes the
updater public key and feed URL into a generated Tauri config written under
`apps/desktop/src-tauri/target/`.

## Sidecar binaries

Production bundles ship three external binaries next to the app executable
(`bundle.externalBin` via `src-tauri/tauri.release.conf.json`):

- `berry-host` — the Node host compiled to a single-file executable with
  Node SEA (esbuild bundle → SEA blob → postject into a copy of the Node
  binary; ad-hoc re-signed on macOS).
- `berry-pty` — the Rust PTY sidecar from `crates/berry-pty`.
- `agent-browser` — the native browser automation runtime installed from the
  `agent-browser` package for the active Tauri target triple.

Build them with:

```bash
node scripts/build-sidecars.mjs
```

The script writes `apps/desktop/src-tauri/binaries/<name>-<target-triple>` and
smoke-tests the host binary with a real `host.handshake` over stdio.
`verify-release-prereqs.mjs --check-sidecars` fails the release if any
binary is missing. Development runs never need them: the Tauri shell prefers
`BERRY_HOST_NODE_ENTRY`/the workspace `packages/host/dist/main.js`, and the
terminal service falls back to plain child processes when `berry-pty` is
absent.

Browser automation uses the `agent-browser` CLI resolved from
`BERRY_BROWSER_CLI`, the packaged `agent-browser` sidecar, or the workspace
`node_modules/.bin` during development. Packaged builds pass the bundled
sidecar to the host through `BERRY_BROWSER_CLI`.

Updater artifacts are enabled for release builds. Generate the signing key with
the Tauri CLI and keep the private key offline except in the release secret:

```bash
pnpm --dir apps/desktop tauri signer generate -w ~/.tauri/berry-updater.key
```

Set `TAURI_SIGNING_PRIVATE_KEY` to the generated private key path or contents,
set `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` if one was used, and set
`TAURI_UPDATER_PUBLIC_KEY` to the public key printed by the signer. The release
workflow runs `scripts/render-tauri-release-config.mjs` before packaging so the
tracked config never stores production key material.

CLI self-update uses a separate Ed25519 key and `berry-cli-update.json`,
generated by `scripts/render-cli-update-manifest.mjs` in the CLI release
workflow. Users can check and stage a verified update with:

```bash
berry update --public-key <keyId=base64>
```
