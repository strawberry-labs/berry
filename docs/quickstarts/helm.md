# Helm

Use the Helm chart for Kubernetes environments that already have ingress, TLS, storage, and secret management.

## Prerequisites

- Kubernetes cluster access with permission to create a namespace, deployments, services, secrets, config maps, and Ingress resources.
- Helm 3.
- External or in-cluster Postgres, Redis, and object storage credentials.
- A DNS name and TLS certificate for the platform/router endpoint.

## Configure values

Start from `deploy/helm/berry-platform/values.yaml` and write an environment-specific override file outside the chart directory:

```sh
cp deploy/helm/berry-platform/values.yaml /tmp/berry-values.yaml
```

Set:

- Image repositories and tags built by the release pipeline.
- Public platform URL and router URL.
- Postgres, Redis, and object storage connection values.
- Managed policy public keys.
- Ingress host, TLS secret, and class name.
- Resource requests and limits for the target cluster.

## Install or upgrade

```sh
helm upgrade --install berry deploy/helm/berry-platform \
  --namespace berry \
  --create-namespace \
  -f /tmp/berry-values.yaml
```

## Confirm it works

```sh
kubectl -n berry get pods
kubectl -n berry get ingress
kubectl -n berry logs deploy/berry-platform-api
curl https://platform.example.test/health
```

- Every deployment is available.
- Ingress serves the configured TLS host.
- The platform health endpoint succeeds.
- Berry desktop can `berry login --base-url https://platform.example.test` and sync managed policy when a public key is supplied.

## Human-owned inputs

Cluster credentials, DNS, TLS, cloud storage, paid provider billing, and production secrets cannot be generated by the repository. Put the final values in the cluster secret manager and the Helm override file used by your deployment pipeline.
